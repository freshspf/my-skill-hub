# 事故案例文件模板

## 输出位置

所有事故案例文件统一放在项目的 `agent_docs/cases/` 目录下。

**初始化时只建空目录，不要预填案例文件，也不要复制模板进去**——案例库要等用户遇到真实事故或明确要求时再写。AI 凭空生成的"通用案例"只会污染上下文。

## 命名规范

`agent_docs/cases/<问题描述>.md`，用 kebab-case，描述问题本质而非 bug 编号。

例：`money-handling-100x-amplification-accident.md`、`pinia-store-naming-collision.md`

---

## 文件模板

```markdown
---
version: v1.0
last_updated: YYYY-MM-DD
---

# [事故标题：简洁描述问题]

**事故级别**: 🔴 P0 / 🟡 P1
**影响范围**: [用户/业务/技术影响描述]
**触发场景**: [什么情况下会遇到这个问题]

## 问题描述

[2-3 句话说清楚发生了什么，为什么会发生。]

## 错误模式

[展示典型的错误代码或操作方式]

\`\`\`typescript
// ❌ 错误：[说明为什么错]
const result = someWrongOperation(value)
\`\`\`

## 正确做法

[展示正确的代码或操作方式]

\`\`\`typescript
// ✅ 正确：[说明为什么对]
const result = someCorrectOperation(value)
\`\`\`

## 强制检查清单

处理相关代码时必须检查：

- [ ] [检查项 1]
- [ ] [检查项 2]
- [ ] [检查项 3]

## 经验教训

[一两句话总结：这个问题的根本原因是什么，如何从架构/流程上避免。]
```

---

## 写法要点

**事故级别定义：**
- 🔴 P0：线上严重事故，有用户损失或核心功能中断
- 🟡 P1：功能异常但有降级方案，或只在特定场景出现
- 🟢 P2：体验问题、样式问题，不影响核心功能

**触发场景要具体：**
- 好：`处理金额/出价/支付相关代码时`
- 差：`处理相关代码时`

**检查清单要可操作：**
- 好：`确认没有对同一金额做两次 * 100 的转换`
- 差：`注意金额处理`

**代码示例要有对比：**
错误和正确的代码并排，让 AI 一眼看出区别在哪。

## 在 AGENTS.md 中引用的格式

```markdown
## 事故案例库

- 🔴 [事故标题](agent_docs/cases/xxx.md) - **[触发时机，即什么情况下必须先看]**
- 🟡 [问题标题](agent_docs/cases/yyy.md) - [触发时机]
```

触发时机是引用的核心价值——AI 需要知道在什么情况下应该主动去读这个文件。没有触发时机的引用形同虚设。
