# PTDD 开发工作流程

端到端的开发工作流技能，从需求说明到代码实施的完整过程。整合头脑风暴和测试驱动开发方法，确保功能设计完善且实施质量可靠。

## 何时应用

在以下场景中**必须**使用此 skill：
- 新功能开发（创建新组件、新模块）
- 功能增强（扩展现有功能）
- 代码重构（改进代码结构）
- Bug 修复（需要测试保障的修复）
- 重大功能变更

## 工作流程概览

| 阶段 | 影响级别 | 规则文件 | 核心要点 |
|------|----------|----------|----------|
| 0. Kickoff | CRITICAL | `rules/kickoff.md` | 项目探索、收集文档链接、确认测试环境 |
| 1. 需求澄清 | CRITICAL | `rules/brainstorm.md` | 一次一问、探索方案、YAGNI |
| 2. 生成计划 | CRITICAL | `rules/plan.md` | 自包含、完整代码、文件结构总览 |
| 3. 确认保存 | HIGH | - | 展示 → 调整 → 保存 → 询问实施 |
| 4. TDD 实施 | HIGH | `rules/tdd.md` | subagent per task、两阶段 review |

## 详细工作流程

### 阶段 0：Kickoff

**在开始前，必须先读取以下规则文件**：
- `rules/kickoff.md`

按 kickoff.md 的三个步骤依次完成：项目探索 → 收集文档链接 → 确认测试环境。

---

### 阶段 1：需求澄清（Brainstorming）

**在开始前，必须先读取以下规则文件，并向用户确认已读取**：
- `rules/brainstorm.md`

**读取后必须输出**："✅ 已读取 Brainstorming 阶段规则，现在开始需求澄清..."

该阶段通过对话方式：
1. 理解项目现状和约束条件
2. 逐步澄清需求细节（**一次一个问题**）
3. 探索并对比 2-3 种技术方案
4. 呈现简洁的设计概要（聚焦高层架构和核心 API）

**禁止行为**：
-  一次问多个问题
-  跳过方案对比直接给出答案
-  在此阶段输出完整实现代码

---

### 阶段 2：生成实施计划

**在开始前，必须先读取以下规则文件，并向用户确认已读取**：
- `rules/plan.md`

**读取后必须输出**："✅ 已读取 Planning 阶段规则，现在开始生成实施计划..."

设计验证通过后，生成详细的实施计划。

计划必须包含：
- 文件结构总览（涉及文件及职责）
- 每个任务的完整测试代码和实现代码（subagent 可独立执行）
- 验收标准

**禁止行为**：
- 代码只写结构不写完整内容
- 跳过测试代码

---

### 阶段 3：展示、确认和保存

#### 步骤 1：直接输出完整计划
- 将生成的计划**完整展示**给用户
- 使用清晰的 Markdown 格式
- 包含所有任务和验收标准

#### 步骤 2：询问是否需要调整
提问："这个计划看起来如何？需要调整吗？"

**如果需要调整**：
- 根据反馈修改计划
- **只重新展示用户觉得有问题的部分**
- 重复此步骤直到用户满意

#### 步骤 3：确认保存路径并保存计划

**保存路径优先级**：
1. 项目 `AGENTS.md` 中有明确约定 → 直接遵循，无需询问
2. 无约定 → 默认路径 `agent_docs/plans/YYYY-MM-DD-<feature-name>-plan.md`，询问用户确认

询问用户（无约定时）："计划将保存到 `agent_docs/plans/YYYY-MM-DD-<feature-name>-plan.md`，是否同意？"

**如果用户同意**：
- 保存到默认路径
- 告知用户计划已保存

**如果用户不同意**：
- 询问用户希望保存到哪个路径
- 根据用户指定的路径保存
- 告知用户计划已保存到指定路径

#### 步骤 4：等待用户明确授权后再实施

计划保存完毕后，提问："方案已确认保存，是否现在开始 TDD 实施？"

**必须等待用户的明确回复**：
- **用户明确同意**（如"开始"、"可以"、"开始实施"等）→ 进入阶段 4（TDD 实施）
- **用户拒绝或无回复** → 停止，结束 ptdd 流程，绝不主动开工

**禁止行为**：
- 不展示计划就直接保存
- 用户未明确确认就擅自开始实施
- 重新展示时输出整个计划
- 把"好的"、"不错"、"没问题"等模糊反馈解读为开始实施的授权

---

### 阶段 4：TDD 实施（可选）

**严格前提：只有收到用户明确的实施指令后，才能进入此阶段。方案写完不等于授权开工，必须重新确认。**

进入阶段后，读取 `rules/tdd.md` 和已保存的 plan 文件，按 subagent per task 流程执行：每个任务 dispatch 独立 `general` subagent，完成后两阶段 review，主线程只做协调。

**如果是 Vue 项目**：遇到 Vue 特定测试问题时，参考 `vue-testing-best-practices` skill。

## 核心原则

### Explore → Main 模式

**需要 explore subagent 的场景**：不知道文件位置、需要大范围搜索（如"项目里有没有用到 X 模式"、"找所有测试文件"）、需要多文件模式匹配的探索型任务。

**主线程直接做的场景**：知道路径的单文件读取（如 `AGENTS.md`、`README.md`）、少量固定的 git/shell 命令、检查某个已知文件是否存在。

区分原则：**探索 = 不确定在哪里、需要搜索；读取 = 知道路径、直接取**。不要为了"读取一个已知文件"而派发 explore subagent——subagent 返回文件内容会污染主线程上下文，得不偿失。

主线程只接收结论摘要（不是原始输出）。测试运行结果只呈现关键数字和结论，不粘贴完整输出。目的是保持主线程对话历史干净，避免中间产物干扰后续推理。

### YAGNI（You Aren't Gonna Need It）

### 增量验证
- 分段展示设计，逐步确认
- 发现问题立即调整
- 保持灵活性

### 测试先行
- 所有生产代码必须先有测试
- 最低 80% 覆盖率
- RED → GREEN → REFACTOR

### 文档驱动
- 所有设计必须先在 plan.md 中记录
- 设计完成后必须显式告知用户评审
- 保持文档与代码同步

---

## 注意事项

1. **不要跳过任何阶段**：这是一套标准的工作流，必须严格遵循
2. **永远不要擅自开工**：写完方案 ≠ 可以开始编码。必须在用户明确授权后才能进入实施阶段，任何模糊的认可都不构成授权
3. **不要主动帮用户提交代码**：commit 由 implementer subagent 负责，主线程不执行 git 操作
4. **保持文档同步**：所有设计变更和实施进度都要及时更新到 plan.md
5. **支持迭代**：允许用户多次补充需求，每次都要重新确认并更新文档

---

**记住**：好的设计来自于充分的理解和探索。花时间做好每个阶段，能够避免大量返工。
